## Фаза 1: Криптографический фундамент

**Цель:** Реализовать ядро криптографии — генерацию ключей X25519, шифрование/дешифрование crypto_box, безопасное хранение приватного ключа через Tink + Android Keystore. Без этого фундамента ни одна другая часть системы не имеет смысла.

**Требования:** CRYPT-01, CRYPT-02, CRYPT-05, CRYPT-06, STOR-04

### Критерии успеха (must_haves)
1. Unit-тест: генерация X25519 ключевой пары через Lazysodium завершается успешно, публичный и приватный ключи имеют длину 32 байта
2. Unit-тест: encrypt -> decrypt round-trip для произвольного текста возвращает исходное сообщение; каждый вызов генерирует уникальный 24-байтный nonce
3. Интеграционный тест: приватный ключ сохраняется через DataStore + Tink, после перезапуска приложения ключ читается и расшифровывает ранее зашифрованные данные

### Зависимости
- Нет (первая фаза)

## Фаза 2: Локальное хранилище

**Цель:** Создать Room-базу данных с таблицами для сообщений, контактов и очереди отправки. Обеспечить персистентное хранение всех данных приложения с корректной схемой и миграциями.

**Требования:** STOR-01, STOR-02, STOR-03

### Критерии успеха (must_haves)
1. Room-схема содержит таблицы: messages, contacts, chats, send_queue — все с корректными индексами и связями
2. DAO-тесты: вставка/чтение/обновление сообщений, контактов и элементов очереди отправки (статусы QUEUED/SENDING/SENT/FAILED) проходят на инструментальных тестах
3. Таблица контактов хранит публичный ключ, отпечаток и статус доверия; запрос по email возвращает корректные данные

### Зависимости
- Фаза 1 (модели данных и формат ключей определены в криптографическом фундаменте)

## Фаза 3: Email-транспорт

**Цель:** Реализовать отправку зашифрованных сообщений через SMTP и получение через IMAP. Определить формат email (MIME-тип, тема, кодировка). Обеспечить дедупликацию и устойчивость к ошибкам.

**Требования:** MSG-01, MSG-02, MSG-03, MSG-04, MSG-05, TRNS-03, TRNS-05

### Критерии успеха (must_haves)
1. Зашифрованное сообщение отправляется через SMTP как email с MIME-типом application/x-cheburmail и темой формата CM/1/<chat-id>/<msg-uuid>, получается через IMAP и расшифровывается на другом аккаунте
2. На стороне получателя создаётся IMAP-папка "CheburMail"; входящие сообщения фильтруются в неё по теме CM/
3. Повторное получение email с тем же UUID не создаёт дубликат сообщения в базе
4. При ошибке SMTP/IMAP применяется экспоненциальный backoff (до 5 попыток); после 5 неудач сообщение помечается как FAILED

### Зависимости
- Фаза 1 (шифрование/дешифрование)
- Фаза 2 (хранение сообщений и очередь)

## Фаза 4: Онбординг и аккаунты

**Цель:** Дать пользователю возможность настроить email-аккаунт (Yandex / Mail.ru) с валидацией подключения. Пошаговый мастер с инструкциями по созданию пароля приложения снижает порог входа.

**Требования:** ACCT-01, ACCT-02, ACCT-03, ACCT-05, UI-04

### Критерии успеха (must_haves)
1. Мастер настройки позволяет выбрать Yandex или Mail.ru; пресеты автоматически заполняют серверы и порты IMAP/SMTP
2. При вводе учётных данных приложение выполняет тестовый IMAP-логин и SMTP-отправку; при неудаче показывается понятная ошибка
3. Пользователь видит пошаговую инструкцию создания пароля приложения для выбранного провайдера (со скриншотами)

### Зависимости
- Фаза 3 (IMAP/SMTP клиент для валидации подключения)

## Фаза 5: Обмен ключами и контакты

**Цель:** Реализовать обмен публичными ключами через QR-код при личной встрече. Пользователь может показать свой QR, отсканировать чужой, увидеть safety number для верификации.

**Требования:** CRYPT-03, CRYPT-04, CRYPT-07, UI-03

### Критерии успеха (must_haves)
1. QR-код содержит публичный ключ + email в стандартизированном формате; после сканирования контакт добавляется в базу с публичным ключом
2. Экран контактов показывает список добавленных контактов; контакт можно просмотреть и удалить
3. Safety number (отпечаток) контакта отображается и совпадает на обоих устройствах для одной и той же пары ключей
4. CameraX-сканер корректно распознаёт QR-код в реальном времени через ML Kit

### Зависимости
- Фаза 1 (ключевая пара для генерации QR)
- Фаза 2 (таблица контактов для хранения)

## Фаза 6: Основной UI чатов

**Цель:** Создать основные экраны мессенджера: список чатов с превью последнего сообщения и экран переписки с историей, полем ввода и индикаторами статуса.

**Требования:** UI-01, UI-02, UI-06

### Критерии успеха (must_haves)
1. Экран списка чатов показывает все активные чаты с именем контакта, превью последнего сообщения и временем
2. Экран чата отображает историю сообщений в хронологическом порядке с полем ввода и кнопкой отправки; отправленное сообщение появляется в списке
3. Каждое сообщение показывает визуальный статус: отправка (часы) -> отправлено (одна галочка) -> доставлено (две галочки) -> ошибка (красный индикатор)

### Зависимости
- Фаза 2 (Room для чтения истории)
- Фаза 3 (транспорт для отправки/получения)
- Фаза 5 (контакты для отображения)

## Фаза 7: Фоновая синхронизация и офлайн

**Цель:** Обеспечить получение сообщений в фоне через IMAP IDLE (foreground service) и WorkManager (fallback). Реализовать офлайн-очередь и уведомления о новых сообщениях.

**Требования:** TRNS-01, TRNS-02, MSG-09, UI-07

### Критерии успеха (must_haves)
1. IMAP IDLE foreground service получает новые сообщения с задержкой менее 5 секунд при активном соединении
2. WorkManager выполняет периодическую синхронизацию каждые 15 минут; сообщения приходят даже после Doze
3. Сообщение, отправленное без сети, ставится в очередь (статус QUEUED) и автоматически отправляется при восстановлении соединения
4. При получении нового сообщения в фоне пользователь видит push-уведомление с именем отправителя и превью (или "Зашифрованное сообщение")

### Зависимости
- Фаза 3 (IMAP/SMTP транспорт)
- Фаза 6 (UI для отображения)

## Фаза 8: Медиа-сообщения

**Цель:** Отправка и получение картинок, файлов и голосовых сообщений. Медиа шифруется crypto_box, передаётся как email-вложение (MIME multipart). Лимит размера — 25MB (ограничение Яндекс/Mail.ru).

**Требования:** MEDIA-01, MEDIA-02, MEDIA-03

### Критерии успеха (must_haves)
1. Пользователь может выбрать изображение из галереи или сделать фото; оно шифруется, отправляется как email-вложение и отображается у получателя как превью с возможностью открыть полноразмерно
2. Пользователь может отправить произвольный файл (до 25MB); получатель видит имя файла, размер и может сохранить на устройство
3. Пользователь может записать голосовое сообщение (длинное нажатие на микрофон); оно кодируется в OGG/Opus, шифруется и отправляется; получатель видит волну и может воспроизвести
4. Прогресс отправки/загрузки медиа отображается в UI (индикатор)

### Зависимости
- Фаза 3 (email-транспорт, MIME multipart)
- Фаза 6 (UI чатов для отображения медиа-баблов)

## Фаза 9: Групповые чаты

**Цель:** Реализовать групповые чаты до 10 участников с fan-out шифрованием (индивидуальный crypto_box для каждого получателя) и управляющими сообщениями для приглашений.

**Требования:** MSG-06, MSG-07, MSG-08

### Критерии успеха (must_haves)
1. Пользователь может создать групповой чат, выбрав до 10 контактов; группа отображается в списке чатов
2. Отправка сообщения в группу генерирует N-1 индивидуально зашифрованных email (по одному на каждого участника)
3. Приглашение в группу рассылается как управляющее сообщение, содержащее публичные ключи всех участников; новый участник видит историю с момента приглашения

### Зависимости
- Фаза 3 (транспорт для fan-out)
- Фаза 5 (контакты и ключи участников)
- Фаза 6 (UI для группового чата)

## Фаза 10: Расширенные функции сообщений

**Цель:** Добавить подтверждение доставки (delivery receipts), исчезающие сообщения с таймером автоудаления и экран настроек приложения.

**Требования:** MSG-10, MSG-11, UI-05

### Критерии успеха (must_haves)
1. При получении сообщения автоматически отправляется зашифрованный ACK-email; отправитель видит статус "доставлено"
2. Пользователь может установить таймер автоудаления для чата (1 час / 1 день / 1 неделя); сообщения удаляются локально по истечении таймера
3. Экран настроек содержит управление аккаунтами, уведомлениями и параметрами исчезающих сообщений

### Зависимости
- Фаза 7 (фоновая синхронизация для отправки ACK)
- Фаза 6 (UI для отображения статусов)

## Фаза 11: Мульти-аккаунт и оптимизация транспорта

**Цель:** Поддержка нескольких email-аккаунтов для увеличения пропускной способности, ротация аккаунтов при отправке, автоматическая очистка старых email на сервере, экспорт/импорт ключей для переноса на другое устройство.

**Требования:** ACCT-04, TRNS-04, TRNS-06, CRYPT-08, CRYPT-09

### Критерии успеха (must_haves)
1. Пользователь может добавить несколько email-аккаунтов; при отправке приложение автоматически выбирает аккаунт с наименьшей нагрузкой (round-robin с учётом rate limit)
2. Обработанные email автоматически удаляются с IMAP-сервера через 7 дней
3. Пользователь может экспортировать ключи в зашифрованный файл (защищённый паролем) и импортировать его на новом устройстве; после импорта старые чаты расшифровываются
4. Ротация аккаунтов при отправке корректно работает — получатель видит сообщения от одного контакта независимо от аккаунта-отправителя
5. Интеграционный тест: при превышении лимита одного аккаунта приложение переключается на следующий без потери сообщений

### Зависимости
- Фаза 4 (настройка аккаунтов)
- Фаза 7 (фоновая синхронизация)
